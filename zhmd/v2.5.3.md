## 已采纳内容（来自 PR #52）

### 1. Bypass 路径 LoRA 损坏修复（`_has_ever_had_loras`）

**文件：** `wrappers/qwenimage.py`（及相关 wrapper）

ComfyUI 的 bypass 可能走入 `loras=[]` 的包装器，却与真正管理 LoRA 的包装器共享同一 transformer。旧的 dirty 检查可能对共享 transformer 调用 `reset_lora_v2`，清空活动 LoRA。bypass 关闭后，管理包装器常因 `_applied_loras` 仍匹配而跳过 recompose，表现为 LoRA 间歇性消失。

**采纳修复：** 单调标志 `_has_ever_had_loras`。只有曾经管理过 LoRA 的包装器才允许 reset / recompose。从未拥有 LoRA 的 bypass 壳不再改写共享模型。

### 2. 从 `IS_CHANGED` 中移除 `model`

**文件：** `nodes/lora/qwenimage.py`、`qwenimage_v3.py`，以及（合并后）`qwenimage_v2.py`

ComfyUI 缓存调用 `IS_CHANGED` 时**不会**传入 MODEL 输入。带 `model` 参数会触发 `TypeError` → ComfyUI 的 `NaN`「始终脏」哨兵 → **即使无变化也每轮 re-compose LoRA**（对 500MB–2GB LoRA 代价很大）。

**采纳修复：** 签名中去掉 `model`；仅对 LoRA 相关控件做哈希（`lora_name`、强度、`cpu_offload`、`save_precompiled_lora` 等）。

### 3. 磁盘持久化 LoRA 预编译缓存

**文件：** 新增 `nunchaku_code/lora_cache.py`；由 `nunchaku_code/lora_qwen.py` 与 LoRA 节点接入

同一 LoRA 文件复用时，不必每代从零 classify + fuse：

- 缓存在 ComfyUI models（`SVDQLora/`）
- 防碰撞路径（父目录哈希）
- 原子写入（`tmp` + `os.replace`）
- 源 LoRA 文件变更时的 mtime sidecar 失效
- B 张量不缩放存储，应用时强度滑条仍然正确

可选 UI：`save_precompiled_lora` 控件（见第 6 项）。

### 4. nunchaku LoRA 应用的 GPU pack / unpack

**文件：** `nunchaku_code/lora_qwen.py`

nunchaku 的 `proj_down` / `proj_up` 相关 `pack_lowrank_weight` / `unpack_lowrank_weight` / `cat` 优先在 Comfy 计算设备（CUDA）上完成，避免每个模块经 CPU paging 往返。reset 路径同样处理。

### 5. `set_lora_strength_v2` 的 `base_rank` 守卫

**文件：** `nunchaku_code/lora_qwen.py`

AWQ modulation 槽位（`img_mod.1` / `txt_mod.1`）没有 `base_rank`，访问 `info["base_rank"]` 会 `KeyError`。无 `base_rank` 的槽位安全跳过。

### 6. 动态 UI 对 `save_precompiled_lora` 的支持

**文件：** `js/z_qwen_lora_dynamic.js`、`js/z_qwen_lora_dynamic_v3.js`

缓存 / 恢复 / 高度计算该控件，避免 redraw / `onConfigure` 后丢失或高度错误。

## 已拒绝内容

### TE V2 CPU offload 加载器（Adoption 4）

**排除：** `nodes/te_offload/` 及其在 `__init__.py` 中的注册。

原因（详见合并计划）：

1. 引用的 Qwen TE 加载器类在已安装的 ComfyUI-nunchaku / ComfyUI 核心中不存在（错误导入目标）。
2. Qwen Image 文本编码器已走 ComfyUI CLIP / `ModelPatcher` offload 路径。
3. 额外的 encode monkey-patch 对本用途没有必要。

---

## 冲突解决 / 覆盖（保留当前 main）

PR 头仍带有 v2.4.x 残留。冲突解决接入后，以下**保持 main 的 v2.5.2 形态**：

| 区域 | 决策 |
|------|------|
| `__init__.py` 版本与节点表 | 保持 2.5.x 版本路径；保留 Krea2 / Diffsynth ControlNet 注册；**不**注册 TE V2 |
| `patches/nunchaku_patch.py` | 保留 rotary `apply_rotary_emb` 兼容 **以及** `suppress_torch_preimport_warning`（v2.5.2） |
| `nodes/lora/qwenimage_v3.py` | 综合 PR 的 `IS_CHANGED` / 预编译 UI 与 main 的日志统一 |
| `.gitignore` | 保留 `.cursor/`、`backups/`、`scripts/` 等仅本地条目 |

main 既有的日志统一（移除 `print()` / Key Diffusion 受 `NUNCHAKU_LOG_ENABLED` 门控）在各节点文件中得到保留。

## 合并后额外修复（`b683f8a`）

合并计划要求、原 PR 头中没有的内容：

1. **在 `compose_loras_v2` 中恢复 `_cached_first_lora_state_dict`**  
   预编译缓存关闭（默认）时，首个 LoRA 不得从磁盘读两次（恢复 v2.2.7 / v2.4.4 的首 LoRA 复用优化）。

2. **`qwenimage_v2.py` 的 `IS_CHANGED`**  
   与 V1/V3 同样去掉 `model`；并停止对 `str(model)` 做哈希（对象 repr 每轮变化也会强制重跑）。

## 对用户的实际影响

- 切换 LoRA 节点 **bypass** 时更稳定
- LoRA 输入未变时重复生成更快（`IS_CHANGED` 不再始终 dirty）
- 可选 **预编译 LoRA 缓存**（大 LoRA 第二代起节省）
- nunchaku pack/unpack 时更少 CPU↔GPU 抖动
- **不会**新增 TE V2 offload 节点（有意拒绝）
- v2.5.0–v2.5.2 功能（Krea2 depth ControlNet LoRA、Diffsynth ControlNet、rotary 兼容、torch 预导入警告抑制）保持不变

## 升级说明

1. 将本自定义节点更新 / 重载到 **v2.5.3**（ComfyUI Manager 读取 `__init__.py` 的 `__version__`）。
2. 更新后完整重启 ComfyUI。
3. 既有工作流可继续使用。支持动态 LoRA 节点上新增可选控件 `save_precompiled_lora`（需要写磁盘缓存时开启）。
4. 若曾在 bypass 切换后出现间歇性 LoRA 丢失，请用本版本复测。
